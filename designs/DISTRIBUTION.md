# Soir Distribution Plan

Plan for turning Soir from "runs from a checkout via `bin/soir`" into an
installable wheel that exposes a `soir` command, covering the C++ extension,
the Python package, the www assets, the sample-pack workflow, and the
free-threaded-interpreter constraint.

No backward-compatibility with the current `SOIR_DIR` / `bin/soir` layout —
this plan describes the end state and the code moves there in one go.

---

## 1. Goals & Non-Goals

### Goals

1. Produce a standard Python wheel (`soir-<ver>-cp314t-cp314t-<platform>.whl`)
   that `pip install` / `uv tool install` places a `soir` command on `PATH`.
2. Require free-threaded Python 3.14t — reject installs on a regular GIL build
   with a clear error.
3. Ship everything needed to run the engine, the www docs site, the TUI, and
   the sample-pack tooling from an installed wheel with no repo checkout.
4. Include a small default sample pack in the wheel (content TBD — design the
   mechanism now).
5. Keep large mutable state (downloaded sample packs, user sessions) out of
   the wheel and under a user data directory.

### Non-Goals

- PyPI publishing. The wheel produced here is PyPI-ready; pushing is out of
  scope.
- `manylinux` policy compliance in phase 1 — first wheels build on a stable
  base image (ubuntu-22.04) and assume glibc ≥ 2.35.

---

## 2. Blockers Today

- **`-Xgil=0` flag.** The console script generated by `[project.scripts]` is
  a plain `python -c "..."` shim. It can't pass interpreter flags. `bin/soir`
  only works because it controls the invocation.
- **`SOIR_DIR` points at the repo.** Engine config (`etc/config.json`), live
  template (`etc/live.default.py`), and sample-pack registry
  (`lib/samples/registry.json`) live *outside* the `soir` Python package and
  aren't in the wheel.
- **No runtime check for free-threaded interpreter.** Wrong-ABI installs
  crash in C++ land rather than erroring cleanly.
- **No multi-platform wheel build.** `just package` tars up a full venv —
  works but is ~500 MB and one-per-host.

---

## 3. Target Architecture

Three directories after these changes:

| Role | Location | In wheel? | Mutable? |
|------|----------|-----------|----------|
| Package code + native `.so` | `soir/` inside site-packages | yes | no |
| Packaged defaults (config, template, registry, default pack) | `soir/_data/` inside site-packages | yes | no |
| User state (installed packs, per-user config, logs, default session) | `$SOIR_HOME` (default: platformdirs user-data dir) | no | yes |
| User session directory (advanced use) | wherever `soir session run <dir>` is invoked | no | yes |

`SOIR_DIR` is gone. `SOIR_HOME` replaces it. If unset, it defaults to
`platformdirs.user_data_dir("soir")`:

- Linux: `~/.local/share/soir`
- macOS: `~/Library/Application Support/soir`
- Windows: `%LOCALAPPDATA%\soir`

New dependency: `platformdirs` (tiny, pure-Python).

---

## 4. Free-threaded Enforcement

Two layers, cheap and redundant on purpose.

### 4a. Wheel tag

Building against free-threaded 3.14 stamps the wheel with the `cp314t` ABI
tag. `pip` / `uv` refuses to install it on a non-free-threaded interpreter.
Nothing to do beyond building on the right Python.

### 4b. Runtime assertion

In the launcher, before anything else:

```python
import sysconfig

if not sysconfig.get_config_var("Py_GIL_DISABLED"):
    sys.stderr.write(
        "Soir requires free-threaded Python 3.14t (the 'nogil' build).\n"
        "Install it with:  uv python install 3.14.2t\n"
    )
    sys.exit(1)
```

`Py_GIL_DISABLED=1` in `sysconfig` means the interpreter was *built*
free-threaded — the invariant we want, independent of the GIL's runtime state.

## 5. The Launcher: replacing `-Xgil=0`

The current `bin/soir` runs `python -Xgil=0 -m soir.app ...`. The flag is
needed because some third-party imports (e.g. `watchdog`) don't advertise
free-threading support and cause CPython to re-enable the GIL at import time.
`-Xgil=0` forces it off.

New entry point — `py/soir/_launcher.py`:

```python
"""Console-script entry point for the `soir` command.

Re-execs under `-Xgil=0` if the GIL is still enabled, then hands off to
`soir.app:app`. Needed because some third-party imports flip the GIL back
on unless the flag is set.
"""

import os
import sys
import sysconfig


def _require_free_threaded() -> None:
    if not sysconfig.get_config_var("Py_GIL_DISABLED"):
        sys.stderr.write(
            "Soir requires free-threaded Python 3.14t.\n"
            "Install it with:  uv python install 3.14.2t\n"
        )
        sys.exit(1)


def _reexec_without_gil() -> None:
    if os.environ.get("_SOIR_GIL_REEXEC") == "1":
        return
    if not sys._is_gil_enabled():
        return
    os.environ["_SOIR_GIL_REEXEC"] = "1"
    os.execv(
        sys.executable,
        [sys.executable, "-Xgil=0", "-m", "soir.app", *sys.argv[1:]],
    )


def main() -> None:
    _require_free_threaded()
    _reexec_without_gil()
    from soir.app import app

    app()
```

`pyproject.toml`:

```toml
[project.scripts]
soir = "soir._launcher:main"
```

Why `os.execv`: by the time the `[project.scripts]` shim has imported
`_launcher`, nothing GIL-unfriendly has loaded yet — but `from soir.app
import app` pulls in typer and the world. Re-execing earns us a clean
interpreter start with the flag applied. The second run hits the sentinel
env var and skips straight through to `app()`.

## 6. Package Resources

### 6a. Layout

`etc/` and `lib/samples/` stay at the repo root — unchanged on disk. The
package exposes them via committed symlinks:

```
py/soir/_data/
├── etc         -> ../../../etc              # symlink
└── lib/
    └── samples -> ../../../../lib/samples   # symlink
```

Editors and existing tooling keep operating on the repo-root paths. Python
code reads via `importlib.resources` and sees a normal package-internal
tree. The wheel builder follows the links and bakes real files into the
archive, so installed wheels are self-contained — the symlinks only exist
in the source checkout.

Default sample pack content lives at its natural home,
`/lib/samples/default/`:

```
lib/samples/
├── registry.json
└── default/                     # default bundled sample pack (TBD content)
    ├── default.pack.json
    └── *.wav
```

`pyproject.toml`:

```toml
[tool.setuptools.package-data]
"soir" = [
    "_data/etc/*",
    "_data/lib/samples/*.json",
    "_data/lib/samples/default/*.json",
    "_data/lib/samples/default/*.wav",
]
"soir.www"  = ["content/*.md", "static/**/*", "templates/*.html"]
"soir.cast" = ["templates/*.html"]
```

Setuptools follows symlinks transparently when expanding globs and building
wheels. Wheel size note: the default pack is the only size-sensitive item
here. Aim for ~5–20 MB.

Windows caveat: a symlinked checkout requires `git config core.symlinks
true` and either Developer Mode or admin privileges. Linux/macOS dev is
unaffected. If Windows dev becomes a near-term need, swap the symlinks for
a build-time `shutil.copytree` hook in `setup.py` populating a gitignored
`_data/` — the `_resources.py` API and `pyproject.toml` globs stay
identical.

### 6b. Resource access API

New module `py/soir/_resources.py`:

```python
"""Access to read-only resources shipped inside the Soir wheel."""

from __future__ import annotations

from importlib.resources import files
from pathlib import Path


def package_data() -> Path:
    return Path(str(files("soir") / "_data"))


def default_config_path() -> Path:
    return package_data() / "etc" / "config.json"


def default_live_template_path() -> Path:
    return package_data() / "etc" / "live.default.py"


def packaged_samples_registry() -> Path:
    return package_data() / "lib" / "samples" / "registry.json"


def packaged_default_pack_dir() -> Path:
    return package_data() / "lib" / "samples" / "default"
```

`importlib.resources.files()` returns a `Traversable`. Wheels unpack to disk
at install time so casting through `Path` is safe. If we ever ship a zipapp,
we switch to `as_file()` context managers.

### 6c. `SOIR_HOME` resolution and first-run bootstrap

`py/soir/config.py` replaces `get_soir_dir()` with:

```python
from platformdirs import user_data_dir


def get_soir_home() -> Path:
    env = os.getenv("SOIR_HOME")
    if env:
        return Path(env)
    return Path(user_data_dir("soir"))


def ensure_soir_home() -> Path:
    home = get_soir_home()
    home.mkdir(parents=True, exist_ok=True)
    os.environ["SOIR_HOME"] = str(home)  # export so the C++ extension sees it

    etc = home / "etc"
    if not etc.exists():
        etc.mkdir()
        shutil.copy(_resources.default_config_path(), etc / "config.json")

    samples = home / "lib" / "samples"
    samples.mkdir(parents=True, exist_ok=True)

    # Seed the default sample pack the first time.
    default_pack_json = samples / "default.pack.json"
    if not default_pack_json.exists():
        src = _resources.packaged_default_pack_dir()
        shutil.copytree(src, samples / "default")
        shutil.copy(src / "default.pack.json", default_pack_json)

    return home
```

Called once at the top of `soir.app:main`. First run copies the default pack
into `$SOIR_HOME/lib/samples/default/` so it behaves identically to any other
installed pack (listed by `soir samples ls`, removable with `soir samples rm
default`, etc.).

### 6d. Updated callers

- `py/soir/config.py::Config.load_global` — reads
  `$SOIR_HOME/etc/config.json`. If the bootstrap has run (always, via
  `ensure_soir_home()` in `app.main`), this file exists.
- `py/soir/cli/samples.py::load_available_packs` — reads registry from
  `_resources.packaged_samples_registry()` (read-only, shipped).
- `py/soir/cli/samples.py::get_installed_packs` — reads
  `$SOIR_HOME/lib/samples/*.pack.json`. The default pack shows up
  automatically after bootstrap.
- `py/soir/cli/session.py` `session mk` — scaffolds using
  `_resources.default_live_template_path()` and `default_config_path()`.
- `py/tests/integration/base.py` — writes `SOIR_HOME` (not `SOIR_DIR`) for
  the isolated temp dir.

### 6e. Path expansion in the engine config

`etc/config.json` ships with `"sample_directory": "./lib/samples"`. That
relative path worked when `SOIR_DIR` was the repo and the engine ran from
it. Once the file is copied into `$SOIR_HOME/etc/`, the string no longer
resolves correctly — the C++ engine (`cpp/core/sample_manager.cc:11`)
opens it relative to the user's CWD, not to `$SOIR_HOME`.

Fix: declare paths with a `$SOIR_HOME` placeholder in the shipped config,
and let the C++ side expand them at read time.

```json
{
    "dsp": {
        "sample_directory": "$SOIR_HOME/lib/samples",
        ...
    }
}
```

The C++ side already has the helper: `cpp/utils/config.cc:89` defines
`Config::ExpandEnvironmentVariables`, a `$VAR`-substitution function
backed by `getenv()`. It is currently dead code — wire it into the
path-typed reads. The load-bearing call site is
`cpp/core/sample_manager.cc:11`:

```cpp
directory_ = utils::Config::ExpandEnvironmentVariables(
    config.GetOrDefault<std::string>("dsp.sample_directory", ""));
```

A short audit of the other path-typed config keys (`live.directory`, any
log-dir setting, etc.) gets the same treatment.

For this to work, `SOIR_HOME` must be visible to the C++ extension via
`getenv()`. Python's `get_soir_home()` only *reads* the env var; the
bootstrap (§6c) also *exports* the resolved value back into `os.environ`
so the engine sees it whether the user set it explicitly or it was
defaulted from `platformdirs`.

Net effect: the wheel ships an unmodified config, the bootstrap copies it
verbatim, and resolution happens in one place at runtime. User-edited
session configs can still override these paths to absolute values — the
placeholder is an authoring convenience, not a requirement.

## 7. The Default Command: `soir` with no arguments

Goal: a user who just ran `uv tool install soir` can type `soir` and
discover the available subcommands.

### 7a. Behavior

- `soir` → print the help message listing available subcommands.
- `soir <subcommand> [...]` → dispatch to subcommand as before
  (`session`, `samples`, `vst`, `www`).
- To start a session: `soir session mk <path>` then `soir session run <path>`
  exactly as today.

### 7b. Root callback wiring

`py/soir/app.py`:

```python
import typer

from soir.cli.samples import app as samples_app
from soir.cli.session import session_app
from soir.cli.vst import vst_app
from soir.cli.www import www_app
from soir.config import ensure_soir_home

app = typer.Typer(no_args_is_help=True)

app.add_typer(samples_app, name="samples")
app.add_typer(session_app, name="session")
app.add_typer(vst_app, name="vst")
app.add_typer(www_app, name="www")


@app.callback()
def main(ctx: typer.Context) -> None:
    ensure_soir_home()


if __name__ == "__main__":
    app()
```

`no_args_is_help=True` makes typer print the help message when no
subcommand is given. The callback still runs `ensure_soir_home()` before
any subcommand invocation so the user-data directory is bootstrapped on
first use.

## 8. Wheel Build

Local build, new justfile recipe:

```make
# Build a wheel for the current platform + interpreter
wheel:
    rm -rf dist/
    uv build --wheel
    @ls -lh dist/
```

Output on Linux x86_64:
`dist/soir-1.0.0-cp314t-cp314t-linux_x86_64.whl`.

The `just package` tarball recipe is removed — superseded by the wheel.

### Multi-platform (phase 2)

`cibuildwheel` in GitHub Actions over a matrix:

- OS: `ubuntu-22.04`, `macos-14` (arm64), `macos-13` (x86_64),
  `windows-2022`
- Python: `cp314t` only

```yaml
name: wheels
on: [push, workflow_dispatch]
jobs:
  build:
    strategy:
      matrix:
        os: [ubuntu-22.04, macos-14, macos-13, windows-2022]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: pypa/cibuildwheel@v2
        env:
          CIBW_BUILD: "cp314t-*"
          CIBW_FREE_THREADED_SUPPORT: "1"
          CIBW_BEFORE_BUILD: "uv pip install pybind11>=3.0.1"
      - uses: actions/upload-artifact@v4
        with:
          name: wheels-${{ matrix.os }}
          path: wheelhouse/*.whl
```

If a given runner can't yet provide `cp314t` through cibuildwheel, fall back
to `actions/setup-python@v5` with `python-version: "3.14t"` and invoke `uv
build` directly.

VST3 SDK and other CMake FetchContent deps will be downloaded per build;
expect ~10 min cold builds per runner.

`manylinux` compliance (via `manylinux` base images or `auditwheel repair`)
is a phase-3 follow-up.

## 9. Distribution Paths

### 9a. Share with a friend (phase 1)

Build locally, hand over one `.whl`. Friend runs:

```bash
uv python install 3.14.2t
uv tool install --python 3.14.2t ./soir-1.0.0-cp314t-cp314t-linux_x86_64.whl
soir
```

`uv tool install` creates an isolated environment for the tool and symlinks
`soir` into `~/.local/bin`. Typing `soir` prints the help message listing
the available subcommands.

### 9b. Bootstrap script (phase 2)

`scripts/install.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail

REPO="yourorg/soir"
VERSION="${SOIR_VERSION:-latest}"

if ! command -v uv >/dev/null 2>&1; then
    echo ">>> installing uv"
    curl -LsSf https://astral.sh/uv/install.sh | sh
    export PATH="$HOME/.local/bin:$PATH"
fi

uv python install 3.14.2t

OS="$(uname -s | tr '[:upper:]' '[:lower:]')"
ARCH="$(uname -m)"
case "$OS-$ARCH" in
    linux-x86_64)   TAG="cp314t-cp314t-linux_x86_64" ;;
    darwin-arm64)   TAG="cp314t-cp314t-macosx_14_0_arm64" ;;
    darwin-x86_64)  TAG="cp314t-cp314t-macosx_13_0_x86_64" ;;
    *) echo "Unsupported platform: $OS-$ARCH"; exit 1 ;;
esac

if [ "$VERSION" = "latest" ]; then
    URL="https://github.com/$REPO/releases/latest/download/soir-$TAG.whl"
else
    URL="https://github.com/$REPO/releases/download/$VERSION/soir-$VERSION-$TAG.whl"
fi

TMP="$(mktemp -d)"
curl -L "$URL" -o "$TMP/soir.whl"
uv tool install --python 3.14.2t --force "$TMP/soir.whl"

echo ">>> soir installed. Run 'soir' to get started."
```

Friend runs:

```bash
curl -sSL https://raw.githubusercontent.com/yourorg/soir/main/scripts/install.sh | sh
soir
```

### 9c. PyPI (phase 3)

Once wheels are manylinux-clean and multi-arch, `uv publish dist/*` puts the
project on PyPI and users get:

```bash
uv tool install soir && soir
```

## 11. Phased Rollout

**Phase 1 — wheel-installable from the repo** (one PR):

1. Add symlinks from the package into the repo-root data dirs:
   ```
   mkdir -p py/soir/_data/lib
   ln -s ../../../etc           py/soir/_data/etc
   ln -s ../../../../lib/samples py/soir/_data/lib/samples
   ```
   Commit both symlinks. `/etc/` and `/lib/samples/` stay at the repo root.
2. Create `_resources.py` and `_launcher.py`.
3. Refactor `config.py` (`get_soir_home`, `ensure_soir_home`).
4. Wire the root typer in `app.py` with `no_args_is_help=True` so `soir`
   with no subcommand prints help; the callback calls `ensure_soir_home()`.
5. Update other callers (`samples.py`, `session.py` subcommands).
6. Switch `etc/config.json` paths to `$SOIR_HOME/...` placeholders and
   wire `Config::ExpandEnvironmentVariables` into the path-typed C++
   config reads (starts in `cpp/core/sample_manager.cc`).
7. Update `pyproject.toml` (script entry, package-data, platformdirs dep).
8. Delete `bin/soir`, delete `just package`, add `just wheel`.
9. Update integration tests to use `SOIR_HOME`.
10. Verify: `just wheel && uv tool install ./dist/*.whl && soir`.

**Phase 2 — hand-off to a friend**:

11. Write `scripts/install.sh`.
12. First tagged release with wheel artifacts uploaded (minimal GH Actions
    job running `just wheel` on ubuntu-22.04 only is enough).
13. README gets the one-liner install instructions.

**Phase 3 — multi-platform & PyPI**:

14. `cibuildwheel` matrix in GH Actions.
15. `auditwheel repair` for manylinux compliance.
16. First PyPI release (`uv publish`).

**Default pack content** lands whenever it's ready — the packaging machinery
is in place from phase 1 onward, so dropping `.wav` files into
`/lib/samples/default/` and rebuilding the wheel is the whole work item.
